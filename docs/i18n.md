# i18n Workflow

This document is the current operational guide for frontend UI translations and
content-language plumbing.

## Source Of Truth

Frontend i18n has three separate layers:

- runtime setup: `web/app/i18n/init.ts`, `web/app/i18n/config.ts`,
  `web/app/i18n/sync.tsx`
- locale JSON: `web/public/locales/{en,zh}/`
- display and API helpers: `web/app/i18n/display/`, `web/app/i18n/hooks/`,
  `web/app/i18n/storage.ts`, and `web/app/api/http.ts`

`web/public/locales/` is the source of truth for UI strings. The
`web/extracted/` directory is generated review scratch space and should not be
edited as the durable locale source.

Chinese spell text and entity names are not normal UI copy. They come from the
app-owned data/import path documented in `docs/import-workflow.md`.

## Changing UI Copy

When adding or changing calls to `t(...)` in frontend code:

1. Use a semantic key in the closest owning namespace.
2. Edit the component key and namespace in `web/app/**`.
3. Run `npm run i18n:sync`.
4. Fill in or adjust English and Chinese values under
   `web/public/locales/{en,zh}/`.
5. Run `npm run i18n:check`.
6. Run `npm run verify` before handoff.

New keys should be stable ids, not English source text:

```tsx
t("nav.browse")
t("search.errors.tooShort")
t("prepared.copy.copiedRows", { count })
```

Locale files remain flat JSON during the migration:

```json
{
  "nav.browse": "Browse",
  "prepared.copy.copiedRows_one": "Copied {{count}} row as TSV.",
  "prepared.copy.copiedRows_other": "Copied {{count}} rows as TSV."
}
```

Do not add new raw-English keys such as
`t("Enter at least 2 characters to run a search.")`. Existing raw-English keys
are tracked in `web/scripts/i18n-legacy-keys.json` only as a migration
allowlist. When a namespace is migrated to semantic keys, remove its legacy
allowlist entries in the same change.

Older raw-English workflow, for reference:

1. Edit the component copy and namespace in `web/app/**`.
2. Run `npm run i18n:sync`.
3. Fill in or adjust Chinese values under `web/public/locales/zh/`.
4. Run `npm run i18n:check`.
5. Run `npm run verify` before handoff.

Useful commands:

```bash
npm run i18n:sync
npm run i18n:check
npm run -w web i18n:audit -- --report
npm run i18n:diff
```

The older `npm run -w web i18next:*` commands remain as compatibility aliases,
but new docs and agent work should use the root `i18n:*` commands.

`npm run i18n:check` runs both extractor drift detection and the local i18n
audit. The audit checks namespace/config consistency, rejects new raw-English
keys outside `web/scripts/i18n-legacy-keys.json`, and compares English/Chinese
locale coverage by normalized plural base key. This avoids false failures for
valid i18next plural differences such as English `_one`/`_other` versus Chinese
`_other`.

## Organization Rules

- Put i18next initialization and namespace defaults in `web/app/i18n/init.ts`
  and `web/app/i18n/config.ts`.
- Put browser preference bridging in `web/app/i18n/storage.ts` and
  `web/app/i18n/sync.tsx`.
- Put pure name/description fallback behavior under `web/app/i18n/display/`.
- Put React composition hooks under `web/app/i18n/hooks/`.
- Keep compatibility re-export files thin when old imports still exist.
- Keep manually maintained data-like namespaces listed in
  `web/i18next.config.ts` `IGNORED`. They are skipped by extraction/sync but
  still audited for locale coverage.

Prefer adding tests for pure fallback helpers when changing display behavior.
The current frontend harness covers storage and display fallback behavior.
