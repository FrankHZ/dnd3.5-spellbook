# i18n Workflow

This document is the current operational guide for frontend UI translations and
content-language plumbing.

## Source Of Truth

Frontend i18n has three separate layers:

- runtime setup: `web/app/i18n/init.ts`, `web/app/i18n/config.ts`,
  `web/app/i18n/sync.tsx`
- locale JSON: `web/public/locales/{en,zh}/`
- display and API helpers: `web/app/i18n/display/`, `web/app/i18n/hooks/`,
  `web/app/i18n/storage.ts`, and `web/app/api/http.ts`

`web/public/locales/` is the source of truth for UI strings. The
`web/extracted/` directory is generated review scratch space and should not be
edited as the durable locale source.

Chinese spell text and entity names are not normal UI copy. They come from the
app-owned data/import path documented in `docs/operations/import-workflow.md`.

## Changing UI Copy

When adding or changing calls to `t(...)` in frontend code:

1. Use a semantic key in the closest owning namespace.
2. Edit the component key and namespace in `web/app/**`.
3. Run `npm run i18n:sync`.
4. Fill in or adjust English and Chinese values under
   `web/public/locales/{en,zh}/`.
5. Run `npm run i18n:check`.
6. Run `npm run verify` before handoff.

New keys should be stable ids, not English source text:

```tsx
t("nav.browse")
t("search.errors.tooShort")
t("prepared.copy.copiedRows", { count })
```

Locale files remain flat JSON during the migration:

```json
{
  "nav.browse": "Browse",
  "prepared.copy.copiedRows_one": "Copied {{count}} row as TSV.",
  "prepared.copy.copiedRows_other": "Copied {{count}} rows as TSV."
}
```

Do not add new raw-English keys such as
`t("Enter at least 2 characters to run a search.")`. Existing raw-English keys
are tracked in `web/scripts/i18n-legacy-keys.json` only as a migration
allowlist. When a namespace is migrated to semantic keys, remove its legacy
allowlist entries in the same change.

Older raw-English workflow, for reference:

1. Edit the component copy and namespace in `web/app/**`.
2. Run `npm run i18n:sync`.
3. Fill in or adjust Chinese values under `web/public/locales/zh/`.
4. Run `npm run i18n:check`.
5. Run `npm run verify` before handoff.

Useful commands:

```bash
npm run i18n:sync
npm run i18n:check
npm run -w web i18n:audit -- --report
npm run i18n:diff
```

The older `npm run -w web i18next:*` commands remain as compatibility aliases,
but new docs and agent work should use the root `i18n:*` commands.

`npm run i18n:check` runs both extractor drift detection and the local i18n
audit. The audit checks namespace/config consistency, rejects new raw-English
keys outside `web/scripts/i18n-legacy-keys.json`, and compares English/Chinese
locale coverage by normalized plural base key. It also requires every plural
base to provide the runtime variants used by each supported locale: English
`_one`/`_other` and Chinese `_other`. This avoids both false failures for valid
cross-locale plural differences and silent fallback from unusable variants. The
audit also treats normalized mechanics labels as a bounded
translation corpus: `spell-filter-vocabulary` mechanics option keys must match
the stable mechanics key exports in `contracts/src/dto/spell.ts`, and Spell
Detail mechanics labels/notes must be present in both English and Chinese.
Missing, placeholder, duplicate, or untranslated Chinese mechanics strings fail
the check. This bounded audit covers the maintained public filter/detail-note
vocabulary; it is not full stat-block mechanics corpus QA.

Full stat-block mechanics localization consumes the data-pipeline display
contract. `SpellMechanicFacet.normalizedText` is a canonical English
equivalence reference and is present only when `displayCoverage` is `complete`.
Localized display should format the structured category/amount/unit/flags for
complete rows. For `partial` or `review`, keep the authoritative raw mechanics
text; do not infer replacement safety from `reviewStatus`. The maintained
complete-display vocabulary lives in the ignored
`spell-mechanic-vocabulary` namespace and is formatted by
`web/app/i18n/display/spell-mechanics.ts`. The i18n audit checks all 40
category, unit, qualifier, flag, punctuation, and template keys in both
languages in addition to the existing filter and detail-note vocabulary.

## Organization Rules

- Put i18next initialization and namespace defaults in `web/app/i18n/init.ts`
  and `web/app/i18n/config.ts`.
- Put browser preference bridging in `web/app/i18n/storage.ts` and
  `web/app/i18n/sync.tsx`.
- Initial language defaults are detected from `navigator.languages` only when
  no saved user preference exists. A saved language toggle always wins over
  browser detection.
- Put pure name/description fallback behavior under `web/app/i18n/display/`.
- Put server-provided spell filter vocabulary display under
  `web/app/i18n/display/spell-filter.ts`. Filter UI should localize accepted
  taxonomy, component, and mechanics keys through this helper and keep the
  server label as the fallback for unknown or future vocabulary. The maintained
  labels live in the ignored `spell-filter-vocabulary` namespace because they
  are data-like server vocabulary keys, not extractor-discovered UI copy.
  Normalized mechanics labels are audited against the shared contracts so
  additions to `castingTimeKeys`, `rangeKeys`, `durationKeys`,
  `savingThrowKeys`, or `spellResistanceKeys` require matching locale strings.
- Put complete Spell Detail mechanics formatting under
  `web/app/i18n/display/spell-mechanics.ts`. It may select normalized display
  only for `displayCoverage = complete`; unsupported categories and every
  `partial` or `review` facet must preserve the raw source field.
- Put React composition hooks under `web/app/i18n/hooks/`.
- Keep compatibility re-export files thin when old imports still exist.
- Keep manually maintained data-like namespaces listed in
  `web/i18next.config.ts` `IGNORED`. They are skipped by extraction/sync but
  still audited for locale coverage.
- Keep user-facing rulebook labels behind `getRulebookDisplay` /
  `useRulebookDisplay`. `rulebook.abbr` remains the source abbreviation,
  `rulebook.displayAbbr` is the optional curated English UI abbreviation, and
  Chinese display prefers localized full names from meta overlays. When a
  Chinese overlay is missing, the full name falls back to English while compact
  display keeps `displayAbbr`, then `abbr`, instead of expanding to that name.

Prefer adding tests for pure fallback helpers when changing display behavior.
The current frontend harness covers storage and display fallback behavior.
