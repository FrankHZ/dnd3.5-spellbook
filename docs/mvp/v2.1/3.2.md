## 3.2 Browse by Level: “All Levels” & Grouped View

### Goal

Enhance level-based browsing to support:

- querying spells across **all levels**
- clearer presentation of multi-level results
- minimal UX controls without changing navigation semantics

This builds directly on the URL-driven browse state from **3.1**.

---

### Scope

#### Backend

- Extend `/api/spells/by-level` to support:
  - `level=all`
  - grouped response by spell level

- Use a **single endpoint** and a **single response shape** for:
  - single-level queries (`level=0..9`)
  - all-level queries (`level=all`)

#### Frontend

- Add an explicit **“All Levels”** option to the level selector
- Support two presentation modes:
  - flat list
  - grouped-by-level list

- Persist browse presentation preferences across sessions

---

### API Behavior

#### Request

`GET /api/spells/by-level`

Query parameters:

- `level`: `0..9` or `"all"` (required)
- `classIds`, `domainIds`: at least one required
- `rulebookIds`: optional (defaults to persisted rulebook scope)
- `page`: 1-based
- `pageSize`: fixed (system constant)

#### Response (always grouped)

```ts
{
  total: number;
  page: number;
  pageSize: number;
  classIds: number[];
  domainIds: number[];
  rulebookIds: number[];
  groups: Array<{
    level: number;
    items: SpellItemView[];
  }>;
}
```

Notes:

- For single-level queries, `groups` contains exactly one group.
- For `level=all`, each `(spell, level)` pair is a distinct result row.
- Results are sorted by:
  - level ascending
  - spell name ascending

---

### Frontend Behavior

#### Level Selection

- Level selector includes:
  - a full-width **“All Levels”** option
  - individual level buttons `0–9`

- Default level is `"all"`
- Level selection is:
  - reflected in the URL (`level=all`)
  - persisted as part of browse defaults

#### Grouped vs Flat View

- A **group toggle** controls presentation only:
  - `flat`: all spells shown in a single list
  - `grouped`: spells grouped by level with level headers

- Grouping does **not** affect:
  - pagination
  - query semantics
  - URL state

#### Card Options

Two browse presentation options are available:

- **Show details** (simple vs full card)
- **Group by level** (flat vs grouped)

These options are:

- persisted in local storage
- not encoded in the URL
- treated as user preferences, not navigation state

---

### Persistence Rules

- Persisted defaults:
  - `browseQuery.level` (default `"all"`)
  - `browseQuery.classIds`, `browseQuery.domainIds`

- Persisted preferences:
  - `cardView`
  - `groupMode`

- URL remains the source of truth for:
  - navigation state
  - pagination

---

### Definition of Done

- `level=all` returns correct, paginated results
- Grouped response works for both single-level and all-level queries
- Frontend can switch between flat and grouped views without re-querying
- Browse preferences persist across reloads
- No breaking changes to 3.1 URL-driven navigation

---

### Notes

- Sorting semantics are intentionally simple and deterministic.
- Grouping is a presentation concern only.
- No attempt is made to deduplicate spells across levels in `level=all` mode.
- This feature is designed to be forward-compatible with richer grouping and sorting in later versions.
