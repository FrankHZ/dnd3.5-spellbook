# i18n Workflow

This document is the current operational guide for frontend UI translations and
content-language plumbing.

## Source Of Truth

Frontend i18n has three separate layers:

- runtime setup: `web/app/i18n/init.ts`, `web/app/i18n/config.ts`,
  `web/app/i18n/sync.tsx`
- locale JSON: `web/public/locales/{en,zh}/`
- display and API helpers: `web/app/i18n/display/`, `web/app/i18n/hooks/`,
  `web/app/i18n/storage.ts`, and `web/app/api/http.ts`

`web/public/locales/` is the source of truth for UI strings. The
`web/extracted/` directory is generated review scratch space and should not be
edited as the durable locale source.

Chinese spell text and entity names are not normal UI copy. They come from the
app-owned data/import path documented in `docs/import-workflow.md`.

## Changing UI Copy

When adding or changing calls to `t(...)` in frontend code:

1. Edit the component copy and namespace in `web/app/**`.
2. Run `npm run i18n:sync`.
3. Fill in or adjust Chinese values under `web/public/locales/zh/`.
4. Run `npm run i18n:check`.
5. Run `npm run verify` before handoff.

Useful commands:

```bash
npm run i18n:sync
npm run i18n:check
npm run i18n:diff
```

The older `npm run -w web i18next:*` commands remain as compatibility aliases,
but new docs and agent work should use the root `i18n:*` commands.

## Organization Rules

- Put i18next initialization and namespace defaults in `web/app/i18n/init.ts`
  and `web/app/i18n/config.ts`.
- Put browser preference bridging in `web/app/i18n/storage.ts` and
  `web/app/i18n/sync.tsx`.
- Put pure name/description fallback behavior under `web/app/i18n/display/`.
- Put React composition hooks under `web/app/i18n/hooks/`.
- Keep compatibility re-export files thin when old imports still exist.

Prefer adding tests for pure fallback helpers when changing display behavior.
The current frontend harness covers storage and display fallback behavior.
